## [CONCERN] phi_s=1.0 / t_lc_s=2.0 参数调优审阅

---

### 1. 安全性评估：phi_s=1.0, t_lc_s=2.0 默认值

结论：CONCERN（有条件接受）

数学分析：
- `phi_s=1.0` 使前向安全阈值为 `D_f_min = phi*v_l + s0 = v_l + 5`，等价于以 1 秒车速为安全余量。这是 IDM/ACC 模型的常见最小值，但低于学界通常推荐的 τ=1.5-2.0s TTC 方法。
- `t_lc_s=2.0`（换道时长 2s）在城市汇入场景中属于中等值。但两者联合降低意味着：系统既允许更小的间隔，又给出更少的缓冲时间。叠加效应未被充分论证。
- 后向约束 `D_r_min = phi*v_c + s0`：v_c 以 CAV 自身速度计算。高速场景（v_c=30 m/s）下 `D_r_min=35m`，而观察到的典型 `g_r=-1.3m`，说明差距来自 L1 车辆密度，与 phi_s 值本身关联不大——这点支持根因诊断，但也意味着 phi_s 降低对实际 gap_reject 率改善有限。

安全风险点：
- 当 fallback 强制合流触发（is_fallback=True）时，`evaluate_merge_point` 完全跳过安全检查（L188-193）。这与 phi_s 值无关，但是存量碰撞风险（Todo 4.2 已有 1 次碰撞记录）。
- phi_s=1.0 默认值若被继承到其他场景（高速/高密度），可能产生安全边界被侵蚀的系统性风险。

建议：phi_s=1.0 可接受，但应在 `MergePointParams` 文档注释中标注其适用场景边界（如最大速度限制），防止参数被盲目复用。

---

### 2. 代码质量审阅

#### 2.1 `_GapDetail` dataclass：APPROVE

设计良好：
- `feasible`、`safety_margin`、`binding_constraint` 三个 property 职责单一、计算正确。
- `margin_f=float('inf')` 默认值正确处理了无前车/无后车的边界情况。
- `binding_constraint` 在 `margin_f == margin_r == float('inf')` 时返回 `'rear'`，属于静默的默认行为——建议在此分支加注释或返回 `'none'` 枚举值以防误解。

#### 2.2 动态属性 `result._best_reject`：CRITICAL CONCERN

```python
# merge_point.py L233
result._best_reject = best_reject # type: ignore[attr-defined]
```

这违反了 dataclass 契约，问题具体如下：

1. 类型系统被绕过：`# type: ignore` 是代码坏味道。任何静态检查工具（mypy/pyright）无法推断 `_best_reject` 的类型，调用侧需用 `getattr(result, '_best_reject', None)` 防御性访问（L450），这是脆弱耦合。
2. 测试覆盖盲区：164 个测试无一直接断言 `_best_reject` 的存在与类型，这是隐患。
3. 修复方案：两种合适选项——
- 方案 A：在 `MergeEvalResult` 中添加字段 `best_reject: '_GapDetail | None' = None`（若希望公开诊断信息）。
- 方案 B：让 `evaluate_merge_point` 返回 `tuple[MergeEvalResult, _GapDetail | None]`，调用侧解包（推荐——强类型，不污染结果对象）。

---

### 3. 根因诊断验证：L1 密度是否真正是瓶颈

结论：诊断基本成立，但论据不完整

支持论据：
- 547 次 gap_reject 中 binding=rear，g_r=-1.3m vs threshold=30m，margin=-31m，说明后车间距严重不足，这直接来源于 L1 主线车流密度。
- 即使 phi_s 降至 0 极端值，threshold_r 变为 `0*v_c + 5 = 5m`，但 g_r=-1.3m 仍无法满足（需 `g_r >= 5m`），gap 本身就是负值，说明 L1 车辆实际已在 CAV 后方更近处，物理上就没有合流窗口。

不完整之处：
- 诊断报告说 action_count=7 bounded by 8 ramp CAVs in 120s，但 lc_issued=3/7，fallback_lc_issued=4/7，说明 4 次合流是 fallback 强制合流而非正常间隙合流。这意味着仿真的【成功合流】中有 57% 是不安全的强制操作。这不是密度单方面的问题，也指向 CAV 速度控制缺失（CAV 没有主动减速/加速以创造间隙的能力）。
- 547 次 reject 中 follow_speed 分布未给出。若 L1 车辆速度离散度高，可能存在部分时段间隙可用但 CAV 速度匹配差的情况。

---

### 4. 改进建议

#### P0 - 必须修复

1. 将 `_best_reject` 纳入 `MergeEvalResult`（或改为 tuple 返回）：消除动态属性，建立类型约束，补充测试覆盖。

2. fallback 路径安全检查：fallback 触发时（L188-193）直接返回 `feasible=True` 不检查任何 gap。鉴于 Todo 4.2 已有 1 次碰撞，建议增加软约束：fallback 时至少检查后向间距是否 > 车身长度（L_veh_m），防止物理碰撞。

#### P1 - 强烈建议

3. `binding_constraint` 当两侧均为 `inf` 时，返回值 `'rear'` 具有误导性。建议返回 `'none'` 或加防御判断。

4. phi_s 场景适用性注释：在 `MergePointParams` 的 `phi_s` 字段添加文档说明其物理含义（TTC 系数），并标明经验验证范围。

#### P2 - 后续优化

5. 速度协调机制：547 次 reject 根因是密度，但 CAV 本身没有主动创造合流机会的机制（减速跟随、加速超越）。这是架构级别的缺失，应在后续 Todo 中列项。

6. `_GapDetail.binding_constraint` 测试：补充边界情况单测（lead=None、follow=None、两者均 None）。

## [REVIEW] Strong A 后接 Todo 4.3 的实验风险评估

核心判断：存在极高风险。极易将“锚点定义/门控宽容度差异”误判为“算法核心差异”。

结论：
1. 执行层掩盖策略差异：Strong A 引入了强力的 Gap 对齐调速与协同让隙，这可能主导了实际合流位置。若执行层对 `expected_merge_position` 仅作软参考，Fixed 和 Flexible 的空间表现会被抹平。
2. 归因谬误（门控时机 vs 算法）：若 Fixed 严格死守固定空间锚点导致高 Fallback，而 Flexible 因滚动更新逃过一劫，这只是“空间宽容度/门控时机”的差异，极易被误当成算法本身的优劣。
3. 协同机制的非对称收益：Flexible 可能因其动态特性更易触发 Strong A 新增的“主路 CAV 协同让隙”，导致指标差异混入了协同机制的副作用，破坏单一变量原则。
4. 契约语义的约束未对齐：目前 `MergeContract` 中的位置/时间窗口，在 Fixed 和 Flexible 下是否受到 MPM 完全同等的硬性约束检验存疑。

最建议的处理方案：
明确并隔离空间门控语义（Spatial Gating Isolation）。在正式跑 Todo 4.3 之前：
1. 检查 `MergePointManager`：确保 Fixed 和 Flexible 走完全相同的 Gap 安全评估与协同触发逻辑。
2. 引入 `spatial_compliance_error` 指标：量化实际合流点与契约预期点的偏差。
3. 剥离测试：建议先在关闭“主路协同让隙”的退化版 Strong A 下跑一次 Fixed vs Flexible，确认纯空间策略差异；再开启协同，观察增量收益，防止机制耦合导致归因错误。
